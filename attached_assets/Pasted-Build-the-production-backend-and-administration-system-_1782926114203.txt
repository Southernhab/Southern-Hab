Build the production backend and administration system for the existing Southern Habitat Consulting website and SHC Landowner Portal.

IMPORTANT

This is no longer a visual mockup task.

Build a real production-capable backend with:

- secure authentication
- database persistence
- role-based access
- client and property administration
- document and image uploads
- ArcGIS map records
- prices, estimates, budgets, invoices, and project costs
- working public inquiry forms
- audit history
- secure deployment configuration

Do not replace or redesign the existing public website.

Do not perform an uncontrolled full-site rewrite.

Create a checkpoint before making changes.

==================================================
ARCHITECTURE
==================================================

Use:

- Existing public website and portal frontend
- Supabase Authentication
- Supabase PostgreSQL
- Supabase Storage
- Supabase Row Level Security
- Supabase Edge Functions where server-side processing is required
- Netlify for frontend deployment
- Environment variables for all credentials

Do not:

- store passwords in the database
- build custom password authentication
- place service-role keys in frontend JavaScript
- hardcode API keys
- expose internal notes through client-facing APIs
- use localStorage as the main database
- use fictional data once production records are enabled

If the existing portal uses a framework, preserve it.

If the existing public website is static HTML, preserve it and keep the portal implementation isolated.

Before coding, inspect and report:

- current framework
- route structure
- portal files
- admin files
- authentication currently present
- any mock-data files
- current Netlify configuration
- current form markup
- current environment-variable support

Then proceed with the implementation.

==================================================
USER ROLES
==================================================

Create these roles:

1. super_admin
2. staff_admin
3. field_staff
4. client_owner
5. client_viewer

Permissions:

super_admin:
- full system access
- manage staff
- manage roles
- manage clients
- manage properties
- manage system settings
- publish client-visible records
- view audit logs

staff_admin:
- manage clients, properties, projects, budgets, reports, maps, and messages
- invite client users
- publish client-visible content
- cannot promote users to super_admin

field_staff:
- add field notes
- upload photographs
- update assigned projects
- add monitoring records
- cannot view unrelated financial information unless explicitly granted
- cannot manage users

client_owner:
- view assigned properties
- view approved client-visible records
- view estimates, budgets, invoices, and files assigned to them
- send messages
- upload requested files
- respond to project decisions
- manage additional client viewers only if staff approval is enabled

client_viewer:
- read-only access to assigned client-visible information
- may send messages if enabled
- cannot approve or defer projects

Protect all permissions in the database with Row Level Security.

Do not rely only on hidden navigation or frontend route checks.

==================================================
ROW LEVEL SECURITY
==================================================

Enable RLS on every production table.

Client users may only access records connected to a client_id assigned to their authenticated user through client_users.

Client users may see only records where:

- client_visible = true
- document_status = published where applicable
- the record belongs to an assigned client/property

Internal notes must never be returned to client users.

Do not solve this by hiding fields in the frontend.

Create safe client-facing database views or RPC functions that omit:

- notes_internal
- internal_notes
- internal_only messages
- draft documents
- unpublished reports
- staff-only financial notes
- audit logs
- unrelated clients

Staff access must also be enforced by RLS.

Service-role operations must run only in secure server-side functions.

Create automated RLS tests.
